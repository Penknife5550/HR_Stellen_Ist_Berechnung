```bash
for m in 0008_deputat_pro_periode 0009_view_pro_tag_range \
         0010_korrektur_layer 0011_view_no_phantom_fill \
         0012_view_no_buffer; do
  sudo docker exec -i stellenist-db psql -U stellenist -d stellenistberechnung \
    < src/db/migrations/${m}.sql

```

Verify:
```sql
-- Sollte 12 Eintraege haben (0000 bis 0012)

-- View muss existieren und Daten liefern
```

## 5. Webhook-Konfiguration

Im UI unter `/admin/n8n-webhooks`:
1. **„Neuer API-Key"** anklicken → Wert kopieren (wird nur einmal angezeigt)
2. Sichtbarkeit: Schluessel beginnt mit `whk_…`

Das schliesst den Bootstrap-Auth-Pfad in `sync-v2` und `untis-terms/sync`:
solange noch keine Webhook-Config existiert, wird `process.env.API_SYNC_KEY`
akzeptiert. Sobald die erste Config angelegt ist, ist nur noch der
DB-gehashte Key gueltig.

## 6. n8n-Workflows aktualisieren

### #223 v0.7 — Sync (Untis → sync-v2)

1. Bestehenden `#223` als Backup exportieren
2. Datei `docs/n8n/#223_v07 - Stellenist Periodenmodell-Sync (Untis - sync-v2).json`
   in n8n importieren (UI → „⋮ → Import from File")
3. **MSSQL-Credential** zuweisen (`Untis`, ID `HzjHGkJk19cln1C7`)
4. Im Code-Node **„Aufbereiten v0.7"** den `apiKey`-Wert auf den frisch
   generierten Webhook-Key aus Schritt 5 setzen (Suchen-Ersetzen im JS)
5. Test-Run: Workflow inaktiv lassen → „Execute Workflow" → pruefen dass
   beide HTTP-Nodes `success: true` zurueckgeben
6. Alten `#223` deaktivieren, `#223 v0.7` aktivieren

**Hinweis zum n8n-Env-Zugriff**: Self-hosted n8n blockt `$env.X`
standardmaessig. Daher wird der Key direkt im Code-Node hartkodiert.
Wenn ihr `N8N_BLOCK_ENV_ACCESS_IN_NODE=false` setzt, kann der Workflow
auf `$env.STELLENIST_API_KEY` umgestellt werden.

### #224 v0.7 — Notifications

1. Datei `docs/n8n/#224_v07 - Stellenist Benachrichtigungen (lehrer + haupt + verteilung).json`
   importieren
2. Outlook OAuth-Credential zuweisen (Platzhalter `DEINE_CREDENTIAL_ID`
   im JSON)
3. Drei Webhook-URLs notieren (n8n zeigt sie nach dem Speichern):
   - `…/webhook/stellenist-lehrer-created`
   - `…/webhook/stellenist-hauptdeputat-changed`
   - `…/webhook/stellenist-verteilung-changed`
4. Workflow aktivieren

### App: Notification-Targets eintragen

Im UI `/admin/n8n-webhooks` (Sektion „Ausgehende Webhooks"):
1. Pro n8n-Webhook-URL einen Target anlegen
2. Beim ersten Target: Events abonnieren — `lehrer.created`,
   `hauptdeputat.changed`, `verteilung.changed`, `sync.completed`,
   `sync.failed`
3. **HMAC-Secret** generieren (in der App) und in n8n als
   Env-Variable `STELLENIST_WEBHOOK_SECRET` setzen
4. „Test"-Button im UI → in n8n sollte eine Test-Mail ankommen

## 9. Periodenmodell-Verhalten

**Was angezeigt wird**:
- Daten enden exakt mit der letzten bekannten `gueltig_bis` aus
  `deputat_pro_periode` (Migration 0012). Beispiel: SY 2025/26 endet
  19.07.2026 → Aug-Dez 2026 sind leer bis SY 2026/27 synct ist.
- Geplante Werte werden nicht verheimlicht: Untis liefert den Jahresplan
  schon im Voraus, daher zeigt z.B. Mai-Juli 2026 Werte (`§ 3 Abs. 1
  FESchVO` — Stellenist basiert auf den geplanten Stunden).

**Periodenmodell-Card** auf Lehrer-Detailseite:
- Echte Wertwechsel werden farblich differenziert:
  - **Rot** = gehaltsrelevant (Gesamt-WS aendert sich) — Badge `GEHALT`
  - **Gelb** = Verteilungswechsel (nur GES/GYM/BK aendert sich) — Badge `VERT.`
- Inline-Edit fuer das tatsaechliche Wirksamkeitsdatum (statt Untis-Montag)
- Plausibilitaetsvalidation: das Datum muss innerhalb der betroffenen
  Periodenspanne liegen, sonst Fehlermeldung

## 10. Rollback-Plan

Falls etwas schiefgeht:

1. App-Container auf vorigen Tag zuruecksetzen:
   ```bash
   sudo docker compose stop app
   sudo docker run --rm -v stellenist_data:/data \
     <previous-image-tag> ...
   ```

2. DB aus Backup wiederherstellen:
   ```bash
   sudo docker exec -i stellenist-db pg_restore -U stellenist -d stellenistberechnung \
     -c -if-exists < /backup/stellenist_pre_v07_*.dump
   ```

3. Migrationen 0008–0012 sind **additiv** (neue Tabellen + Views, alte
   Tabellen unangetastet). Daher reicht im Notfall ein App-Rollback +
   Beibehalten der neuen DB-Objekte. Die `deputat_monatlich`-Tabelle ist
   nach v0.7 leer; ein Rollback der App auf v0.6 wuerde sie aber wieder
   befuellen, sobald der alte Sync laeuft.
